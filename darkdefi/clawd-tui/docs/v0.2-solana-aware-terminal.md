# Clawd v0.2 — A Solana-Aware Terminal

> 🦞 Paste a contract. Get a chart's worth of context before the agent ever wakes up.

Clawd v0.1 was a solid agent terminal — adaptive input block, streaming tools, OAuth onboarding, session persistence. Useful, but every other agent terminal had the same shape. v0.2 makes Clawd the **first** Solana-native one.

This release adds a Birdeye client, a Helius DAS client, eleven new slash commands, and an on-paste detector that turns any base58 mint or wallet address into an inline market + on-chain card — without spending an agent step.

---

## Why this matters

Working with Solana is a constant context-switch:

> *"What's that token's liquidity? Is this address a wallet or a mint? Is this NFT compressed? Who holds the supply? What's my net worth right now?"*

Each question normally means: tab to a browser, paste, wait for a UI to load, copy a number back. Doing that mid-coding-session breaks flow. Doing it inside an agent loop is *expensive* — every lookup costs steps and tokens.

Clawd v0.2 collapses all of it into the prompt itself.

## What's new

### Two new API integrations

| Provider     | Endpoint family                        | What we use it for                                         |
| ------------ | -------------------------------------- | ---------------------------------------------------------- |
| **Birdeye**  | `/defi/*`, `/defi/v3/*`, `/v1/wallet/*` | Token overview, trending, search, wallet portfolio + worth |
| **Helius**   | DAS JSON-RPC + Solana RPC              | Single-asset lookup, owner/NFT search, supply, holders, signatures, native SOL |

Both clients live in pure TypeScript with no runtime dependencies — just `fetch`, an `AbortController` timeout, and tight types pulled from the official Birdeye and Helius docs.

### Eleven new slash commands

```text
Birdeye (market data)
  /trending [n]            top trending Solana tokens
  /search   <query>        keyword search, ranked by 24h volume
  /wallet   <addr>         top USD holdings for a wallet
  /portfolio <addr>        full token portfolio with values
  /networth <addr>         total USD net worth + top-5 weights

Helius (on-chain)
  /asset    <id>           single asset/NFT/token via DAS getAsset
  /assets   <addr> [page]  full DAS holdings (NFTs + tokens + SOL)
  /nfts     <addr>         NFTs only (regular + compressed)
  /holders  <mint>         supply + top accounts with concentration colors
  /sigs     <id>           recent signatures for a compressed asset
  /balance  <addr>         native SOL balance
```

Each one prints a hand-tuned card in the lobster palette — orange symbols, dim labels, green/red percentage diffs, and concentration colors that flip yellow at 1% and red at 10%.

### Auto-analysis on paste

The headline feature. Paste any base58 32–44-char string into the prompt:

```
> So11111111111111111111111111111111111111112
```

Clawd recognizes it as a Solana address, fans Birdeye + Helius out in parallel, prints both cards, and returns to the prompt. **The agent is never invoked.** No tokens spent, no step count consumed, no waiting on an LLM round-trip.

The two providers complement each other:

- **Pure SPL token mint** → Birdeye prints price/mcap/liquidity/volume; Helius silently no-ops because it would just duplicate Birdeye's price data.
- **NFT or compressed asset** → Helius prints the DAS card (interface, compression, creators, collection, royalty); Birdeye usually no-ops because NFTs aren't on its index.
- **Wallet address** → Both attempt to resolve. Helius silently no-ops if it's not also a registered asset.
- **Unknown / new mint** → Helius surfaces on-chain supply even when Birdeye hasn't indexed it yet.

You don't have to think about which one will fire. Set whichever keys you have and Clawd routes around the gaps.

## Architecture

Five new files, ~700 lines total, no runtime deps added:

```
src/
├── birdeye.ts          REST client + types (token/trending/search/wallet)
├── helius.ts           DAS + Solana RPC JSON-RPC client + types
├── format-token.ts     Birdeye TUI cards (overview, trending table, portfolio)
├── format-helius.ts    Helius TUI cards (asset, owner split, holders, sigs)
└── commands.ts         Slash registry + maybeAnalyzeAddress paste hook
```

The CLI loop ([src/cli.ts](../src/cli.ts)) gained two lines:

```ts
if (await maybeAnalyzeAddress(trimmed, ctx)) {
  continue;
}
```

If the input parses as a Solana address and at least one provider is configured, `maybeAnalyzeAddress` prints whatever it can and skips the agent call entirely. Otherwise it returns `false` and the prompt flows to the agent normally.

### `maybeAnalyzeAddress` — the parallel fan-out

```ts
const tasks: Promise<unknown>[] = [];
let birdeyeOk = false;

if (ctx.config.birdeyeApiKey) {
  tasks.push(
    bird.tokenOverview(addr)
      .then(d => { birdeyeOk = true; print(formatTokenOverview(d)); })
      .catch(swallow404),
  );
}

if (ctx.config.heliusApiKey) {
  tasks.push(
    hel.getAsset(addr)
      .then(a => {
        const isFungible = a.interface === 'FungibleToken' || a.interface === 'FungibleAsset';
        if (!birdeyeOk || !isFungible) print(formatAsset(a));
      })
      .catch(swallow),  // wallets aren't DAS assets — silent fall-through
  );
}

await Promise.all(tasks);
```

Errors are swallowed into a single tasteful "not indexed" message, never a stack trace. The Helius dedup check (`!birdeyeOk || !isFungible`) is what keeps you from seeing two price quotes for `So111…112`.

### Why no SDK?

Both vendors ship official SDKs. We didn't reach for them because:

1. Clawd is a TUI — every dependency is bytes a user has to install before their first prompt.
2. The endpoints we use are stable JSON over HTTPS. A 200-line typed `fetch` wrapper costs less than the SDK's transitive deps.
3. We get to spell types exactly the way the rest of the codebase wants them, instead of unwrapping the SDK's nested envelopes.

If you need an endpoint we don't expose yet, [src/birdeye.ts](../src/birdeye.ts) and [src/helius.ts](../src/helius.ts) are 200 lines apiece — adding a method is a 5-line PR.

## Setup

Two env vars. Both optional. Either one unlocks the relevant features.

```bash
# .env
BIRDEYE_API_KEY=...      # bds.birdeye.so
HELIUS_API_KEY=...       # helius.dev
HELIUS_RPC_URL=...       # optional override
```

That's it. Restart `clawd` and the new commands light up. Without the keys, the commands print a friendly `error: HELIUS_API_KEY is not set` instead of crashing — and the on-paste detector silently falls through to the agent so nothing breaks.

## What this enables

A few flows that used to be a tab-switching slog:

**Quick token check before a buy.** Paste the mint, get price/liquidity/24h volume + on-chain supply in under a second. Decide. Continue coding.

**"Wait, what's in this wallet?"** `/portfolio <addr>` returns a sorted USD-value table. `/networth <addr>` collapses it to a one-line total + top-5 weights.

**NFT triage.** Paste a mint — Helius tells you if it's compressed, who the verified creators are, what collection it belongs to, the royalty. Useful when sniffing through an unknown drop.

**Holder concentration.** `/holders <mint>` shows top-10 accounts with percentages colored by concentration. >10% glows red. Instant rug-tell.

**Compressed asset history.** `/sigs <id>` pulls the on-chain transaction history for a cNFT — something most NFT viewers don't surface at all.

**Trending hunt.** `/trending 25` is faster than opening Birdeye's homepage.

## What's next

A few directions we're eyeing for v0.3:

- **Holder overlap** — given two mints, how many wallets hold both? (Helius `getTokenAccounts` + intersection.)
- **Live websocket panes** — stream new pairs / large transactions from Birdeye's WSS into a side pane while the agent runs.
- **`/swap` simulation** — Jupiter quote + slippage preview from a slash command, gated behind explicit approval (no live execution by default).
- **NFT image preview** — iTerm2 / Kitty inline-image protocol for `/asset` cards on a compressed NFT.

If you'd use any of these, [open an issue](https://github.com/clawdsolana/OpenClawd/issues) — that's how the v0.2 command set got prioritized.
